## 二、关键决定及原因

### 决定 1：Docker 用 `node --import tsx/esm` 而不是编译 TypeScript

**背景**：`packages/shared` 直接导出 `.ts` 源文件，不预编译。Dockerfile 里如果只编译 `apps/backend`，`tsc` 找不到 shared 的 `.js` 输出。  
**选择**：用 tsx ESM 运行时（`node --import tsx/esm src/index.ts`），在 Node.js 22 运行时动态编译 TS。  
**代价**：每次启动多 ~0.5s，生产冷启动稍慢。对 Fly.io 单实例 MVP 可接受。

### 决定 2：bcryptjs 替换 bcrypt

**背景**：bcrypt 依赖 `@mapbox/node-pre-gyp`，后者依赖 `tar@^6`，tar@6 有未修复的 CVE。pnpm overrides 无法跨 major 版本解决（tar@7 的 patch 与 @mapbox 的 `^6` 要求冲突）。  
**选择**：换用 `bcryptjs`（纯 JS，API 完全相同，无原生依赖）。  
**代价**：略慢（无 C++ 绑定），cost=12 仍在 ~250ms 范围内，OWASP 合规。

### 决定 3：drizzle-orm 从 0.33 升级到 0.45

**背景**：drizzle-orm 0.33.0 有 SQL 注入 CVE，CI 安全审计（`pnpm audit --audit-level=high`）直接报红。  
**选择**：升到 ^0.45.2（最新稳定版）。  
**代价**：drizzle-orm 0.45+ 把 PG 错误包在 `DrizzleQueryError.cause` 里，之前用 `err.code === '23505'` 检测 unique violation 的代码全部失效。提取了 `lib/pg-errors.ts` 的 `isPgUniqueViolation` 递归检查函数统一修复。

### 决定 4：限流用 in-memory 而不是 Redis

**背景**：MVP 阶段用户量极少（pre-launch），引入 Redis 增加运维复杂度和成本。  
**选择**：在 `rate-limit.ts` 里用 `Map` 做 module-level 单例，`setInterval(...).unref()` 定期清理过期桶。  
**代价**：多实例水平扩展时限流不准确（每台机器独立计数）。Fly.io 目前 min\_machines=0、max 1 台，不存在多实例问题。升到 Redis 的时机：DAU > 500 或开始水平扩展时。

### 决定 5：LLM 代理配额在调用前原子扣减，失败时补偿

**背景**：先扣再调可防并发滥用，但 vendor 503 时用户白白损失一次配额，体验差。  
**选择**：先原子扣减（`UPDATE ... WHERE coldStartQuotaUsed < 100`），vendor 失败时回补（再 +1）。两步操作不原子，极端情况（补偿失败）用户损失 1 次，但 100 次免费额度内可接受。  
**未来**：如果用户投诉，改为事务内扣减 + 提交前确认 LLM 响应（需重构为流式）。

### 决定 6：logout 幂等且永远返回 200

**背景**：如果 logout 对"token 不存在"返回 404，攻击者可枚举 refresh token 是否有效。  
**选择**：`revokeRefreshToken` 只做 `UPDATE WHERE token = $1`，不查是否存在，0 行匹配也不报错，路由永远返回 `{ loggedOut: true }`。  
**代价**：无，这就是正确做法。

### 决定 7：`POST /v1/auth/logout` 不需要 access token

**背景**：用户退出时 access token 可能已过期（1h TTL），如果 logout 要求有效 access token，用户在"token 过期"状态下无法主动登出。  
**选择**：logout 只验证 refreshToken body 格式，不走 `requireAuth` 中间件。  
**代价**：任何人只要知道 token 字符串就能撤销它——但这正是设计意图，不是缺陷。
